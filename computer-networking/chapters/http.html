HTTP/2 and HTTP/3

HTTP/2 removes HTTP-layer head-of-line blocking between streams (no more “request 1 blocks request 2” at the framing layer). Those streams are still serialized into one TCP connection, so transport-level HoL remains: one lost segment creates a gap in the shared byte sequence, and TCP cannot hand later bytes to the application until the hole is filled—those later bytes may carry a different stream’s frames. HTTP/3 over QUIC maps streams to separate QUIC streams with independent ordering and loss recovery, so a loss on stream A does not force the same stall pattern on B and C (you still share congestion control and path limits).

Head-of-line: where the bottleneck sits

HTTP/2 + TCP

Streams A B C  (HTTP frames interleaved)
||||||||||||  (one TCP sequence: bytes 1…n)
     ^ lost segment
──── block until retransmit fills the hole
→ multiplexing helps HTTP, not TCP’s HoL

HTTP/3 + QUIC

Stream A  ───●── gap ───→  (repair A)
Stream B  ───────────────→  (keeps moving)
Stream C  ───────────────→
→ loss still costs capacity; HoL shape differs

Mnemonic: H2 fixes “six connections” pain; TCP can still serialize delivery after loss. H3 moves byte-stream ownership so stream-level stalls decouple more cleanly—without removing congestion signals or server work.

HoL at HTTP (framing)
Largely gone in both: interleaved streams avoid the old “queue behind response 1” problem.
HoL at TCP (single ordered byte pipe)
Still affects H2: one gap in the sequence stalls everything after it. Not applicable to QUIC in the same way because delivery order is per-stream.
HoL-shaped stalls under HTTP/3
Different profile: shared congestion controller, CPU, and application queues can still serialize work—but the classic “lost TCP segment blocks unrelated streams’ bytes” pattern is what H3/QUIC is designed to reduce.

What multiplexing fixes
  • Many small assets no longer require six parallel TCP connections to avoid self-HoL.
  • Server push existed in H2 but is rarely the first lever—prioritization and frame scheduling matter more.
What still hurts

Application-level queues, database contention, and TLS/crypto still cap QPS. The explorer’s RTT/loss controls are a reminder that transport remains the floor for interactive workloads.
