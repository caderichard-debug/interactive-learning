Topic 03 · fairness

Rate limiting algorithms

Limits protect origin capacity and isolate noisy neighbors. Fixed windows are simple but admit burst seams; token buckets and leaky buckets shape sustained vs burst traffic more smoothly.

Side-by-side intuition

Fixed window counters reset on a clock tick. Two heavy bursts that straddle the boundary can temporarily consume twice the nominal per-window budget—fair for some batch jobs, surprising for others—so document the behavior and return predictable 429 + Retry-After when possible. Token bucket refills continuously up to a capacity, so short bursts borrow from the bucket while long-run throughput tracks the refill rate.

Algorithm intuition
  • Fixed window: count requests per clock bucket; easy, but two bursts across the boundary can temporarily allow 2× the nominal rate.
  • Token bucket: refill tokens continuously up to capacity; admits bursts up to capacity while bounding long-run rate to refill.
  • Sliding log / sliding window: more accurate, higher memory cost—use for strict SLOs or billing.
Client-facing behavior

Return 429 with Retry-After when you can predict recovery. Pair with jittered exponential backoff on clients to avoid synchronized retry storms after incidents. Expose remaining quota and reset epoch in headers when you can (RateLimit-*, X-RateLimit-*, or similar vendor-specific shapes)—clients integrate faster when counters are inspectable.
