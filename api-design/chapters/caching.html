Caching and staleness

Caches move work from the hot path to a stored answer. The design question is not “if” but what can be wrong for how long—your staleness budget—and how you purge when reality changes.

Keys, TTLs, and purge groups

Anything that changes semantics belongs in the cache key: tenant, locale, feature flags, and representation version. TTL alone is a blunt instrument; surrogate keys (tag-based purge) let you invalidate families of responses when an underlying entity changes—critical for product catalogs and CMS-driven pages.

Wire shape interplay

REST-shaped GET traffic is the happy path for shared CDNs. GraphQL and authenticated POSTs often move caching to application or edge rules you control. When you model latency, separate cache luck (hit rate) from origin work (chatty fetches vs one hop)—the explorer’s panel 2 makes that split explicit; see REST vs GraphQL for the operational context.

Key design choices
  • Key shape: include versioning, tenant, locale, and anything that changes the semantic response.
  • TTL + max-age: start conservative; measure hit ratio and origin error budget together.
  • Invalidation: surrogate keys (Fastly-style) or tag-based purge beat ad-hoc URL lists for grouped content.
  • Conditional requests: ETag / If-None-Match saves bandwidth when validators are cheap.
Failure modes

Thundering herd after expiry—use jittered TTLs, stale-while-revalidate, or request coalescing. Poisoned entries from error responses—never cache 5xx without tight bounds; consider negative caching only with care.

Production defaults
  • Instrument hit ratio, origin latency, and purge success; alert when hit ratio drops without a matching traffic change.
  • Prefer explicit Cache-Control over implicit behavior; document what must never be stored (PII-heavy payloads).
  • Pair with Auth & webhooks when signed URLs or tokens participate in cache keys.
