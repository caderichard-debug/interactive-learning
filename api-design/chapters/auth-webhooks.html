Topic 05 · trust boundaries

Auth and webhooks

Public APIs and inbound webhooks share a theme: prove caller identity, scope power narrowly, and assume duplicates will arrive.

Tokens for outward APIs

Short-lived access tokens limit blast radius. Refresh flows should rotate refresh tokens and detect reuse, which often indicates leakage. Encode audience and scopes explicitly so a token minted for tooling cannot be replayed against customer data paths by accident.

Webhooks as untrusted input

Treat payloads like any user input: validate shape, verify signatures with constant-time compares, and reject stale timestamps outside a tight replay window. Your handler should be idempotent on a stable event identifier so at-least-once delivery does not create duplicate side effects.

Tokens (OAuth-style APIs)
  • Prefer short-lived access tokens + refresh with rotation and reuse detection.
  • Encode audience and scopes explicitly; reject cross-service token confusion.
  • Log jti or token fingerprint hashes—not raw secrets—for incident response.
Webhooks
  • HMAC signatures over body + timestamp; verify with constant-time comparison.
  • Replay window: reject timestamps older than a few minutes; use monotonic nonces when the vendor supports them.
  • Idempotency keys on your side: same event ID → same outcome, safe under at-least-once delivery.
Checklist before go-live

mTLS or signed URLs where appropriate · rotate signing secrets · bounded retry backoff on your dispatcher · DLQ for poison messages · alert on signature failure spikes · document how cache keys incorporate auth state so you never serve one tenant’s data to another
