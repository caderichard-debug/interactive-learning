REST vs GraphQL

REST maps resources to URLs and verbs; clients stitch multiple responses for nested screens. GraphQL exposes one typed graph: clients declare fields, servers resolve a tree—often in one HTTP round trip.

Payload fit and chatty UIs

REST can over-fetch per resource while still under-delivering for a screen that needs a graph of related entities—hence the classic 1 + N pattern when clients walk links. GraphQL can target the screen shape, but under-modeled graphs or permissive schemas invite huge queries unless you enforce budgets.

Caching and intermediaries

Cache-friendly GETs with stable URLs align with shared CDNs. Typical GraphQL transports are harder to cache on shared edges; teams lean on persisted queries, private edge rules, or normalized client caches. Pair this topic with Caching when you design TTLs and purge paths.

Evolution and operations

REST evolution often means additive JSON fields or versioned paths. GraphQL evolution is schema-first: deprecations, field analytics, and breaking-change discipline in a registry. Pair breaking-change policy with explicit compatibility windows—see Versioning. Operationally, GraphQL needs resolver-level tracing, depth and complexity limits, and timeouts—cost controls are not optional on the public internet.

Trade-offs (compressed)
Topic REST GraphQL
Payload fit Risk of over-fetch per resource; chatty when the UI needs many related entities. Field selection reduces over-fetch; risk of under-modeled graphs and huge queries.
Caching HTTP caches and CDNs align naturally with URL-addressed GETs. POST-shaped endpoints complicate shared caches; rely on persisted queries, APQ, or private edge rules.
Evolution Version in path or header; additive JSON fields are common. Schema deprecation, @deprecated, and breaking changes need registry discipline.
Ops Predictable per-route capacity; simpler mental model for many teams. Needs query cost analysis, timeouts, depth limits, and resolver-level observability.
Practical defaults
  • Prefer BFFs or server-driven aggregates when the client is owned by the same team as the API.
  • Use persisted queries or strict complexity budgets before exposing arbitrary GraphQL on the public internet.
  • Instrument N+1 resolver patterns; batch loaders are table stakes at scale.
  • Document rate limits per operation class; pair with the Rate limits chapter when you expose graphs to third parties.
