Versioning and compatibility

APIs live longer than any single client. The sustainable default is additive change first: new optional fields, new endpoints, and conservative defaults—then explicit deprecation with timeboxed sunsets.

Additive-first discipline

Prefer changes consumers can ignore: optional fields, expanded enums with safe fallbacks, and new resources instead of semantic rewrites. When you must break, ship a parallel surface (/v2 or a new resource) and migrate traffic with metrics on old vs new paths—see REST vs GraphQL for how schema registries change the playbook.

Deprecation windows

Publish a calendar date, log usage by consumer key, and send staged warnings (headers, docs, email for partners). Pair contract tests with traffic dashboards so you retire fields only when usage crosses a safe threshold. In GraphQL registries, @deprecated plus field analytics gives concrete removal signals; in OpenAPI, mark fields deprecated with a sunset note in the description or extension fields.

Patterns
Approach When it shines Watch out
URL prefix (/v2/) Clear routing, easy docs and analytics split. Proliferation of duplicate handlers if not generated from one spec.
Header negotiation Stable URLs; good for incremental media types. Caches and intermediaries must vary correctly on headers.
Schema registries (protobuf, Avro) Strong contracts for internal high-throughput services. Consumer upgrade coordination; compatibility matrices.
Contract tests

Capture consumer-driven contracts or golden fixtures for critical clients. Break the build when a server change violates agreed JSON shapes or OpenAPI examples—before traffic shifts.

Production defaults
  • Never remove a field without a deprecation header, doc notice, and a dated sunset.
  • Generate server stubs and client SDKs from the same OpenAPI/proto source where possible.
  • Alert on rising 4xx after releases—often the first signal of a compatibility miss.
